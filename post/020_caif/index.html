Cache Types

1) In-Memory Cache

This uses in-memory data structure to implement a cache, such as ConcurrentHashMap in Java. If we need other functionalities such as eviction policy, then ConcurrentHashMap won’t provide a convenient way to do that. This is where a library like Caffeine comes in.

Advantages

  • Super fast.

Disadvantages

  • Data is local to the service instance.
  • Limitted by the memory available to the service instance.

2) Distributed Cache

Distributed cache typically stores data in memory across multiple nodes or servers in order to scale horizontally. Examples include Redis, Couchbase, Memcached, etc.

So why not just cache in the memory of the service instance?

Advantages

  • If cache on service instance, the data will be wiped out every time when the service gets a deployment.
  • Can scale cache independently.
  • Request coalescing. If multiple instances request the same data, distributed cache can deduplicate multiple requests that would have been sent to the backing data store.

Disadvantages

  • Mode complex setup.

3) Client-Side Caching

Advantages

  • Caching on the client side can help reduce server requests

Disadvantages

  • It complicates the analytics as the backend will not receive the request. Will need additional logging in the client to collect metrics.

Caching Strategies

Read Strategy

1) Read Aside

Advantages

  • Greate for read-heavy loads
  • Only requested data is written to cache

Disadvantages

  • Cache data can be inconsistent if writes are made to database directly.
  • Unavoidable overhead when cache miss (Can mitigate to some extent by cache warming).

2) Read Through

Slight different, where the responsibility of the DB read is shifted to the cache.

Write Strategy

1) Write Through

Advantages

  • Data is consistent.

Disadvantages

  • Slower writes; every single write goes to both cahce & database.
  • Some data written to cache is never read; unnecessary cost (Can mitigate by setting TTL to reduce space).

2) Write Back

Advantages

  • Faster writes; writes are only made to cache initially. The writes to the DB is postponed until the modified content is about to be evicted or other cache invalidation policy.

Disadvantages

  • More complex.
  • In case of cache failure, the data may be permanently lost.

3) Write Around

In this approach, the application only writes to the DB.
